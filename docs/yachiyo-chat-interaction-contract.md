# Yachiyo Chat Interaction Brief

## 这份文档要解决什么

Yachiyo 现在已经跑通了基础 AI chat、database 重构和 model selector，说明主链路已经成立。

当前最需要补的，不是再加一堆新能力，而是把 chat 交互本身打磨得更像一个完整产品。

问题不在“能不能答”，而在“答的时候用户感受到什么”。

现在的粗糙感主要来自这几件事：

- 用户发出消息后，系统虽然在工作，但缺少清晰、自然的阶段反馈
- model selector 虽然已经存在，但还没有真正成为聊天体验的一部分
- 中间过程几乎都被压扁成一个模糊的 `Thinking...`
- 回答完成、失败、停止，这几种结束状态还不够有区分度
- 整体交互更像一个能跑的壳，而不是一个有节奏、有秩序的 chat 产品

一句话：Yachiyo 下一步的重点应该是 UX，而不是继续堆 capability。

## 总方向

Yachiyo 应该是一个 `chat-first, agent-ready` 的产品。

这句话在 UX 上的含义是：

- 用户首先感受到的是自然、顺滑、可信的聊天体验
- 系统内部即使存在更复杂的 runtime、harness、plugin、memory 过程，也不应该直接把这些技术结构生硬地甩到脸上
- 但这些过程也不能完全消失，因为一旦完全消失，用户就会觉得“系统在黑箱里突然想完了”

所以目标不是“把所有内部细节可视化”，而是：

- 让用户知道系统正在做事
- 让用户大致理解系统现在处于哪个阶段
- 让用户知道这次回答是怎么收尾的
- 让用户在切模型、重试、停止时有明确预期

## UX 优先级

### 1. 先解决状态感

现在最该补的是“状态感”。

用户发送一条消息后，至少应该能自然感知到：

- 消息已经被接收
- 系统正在准备这次回答
- 系统正在输出回答
- 如果中间发生了额外处理，用户能看到轻量提示
- 最后这次回答是正常完成、失败，还是被手动停止

这里的关键不是做复杂动画，而是做明确、克制、连续的反馈。

如果这层做对了，哪怕能力还不多，产品也会立刻显得成熟很多。

### 2. 让 model selector 成为 chat 的一部分

现在 model selector 已经接上了，这是很好的节点。

接下来不应该把它继续当作一个“孤零零的控制项”，而应该把它变成用户理解当前对话的一部分。

用户至少应该明确：

- 当前默认会用哪个模型
- 这次回答实际上用了哪个模型
- 切换模型影响的是接下来的回答，而不是正在生成中的这次回答

重点不是炫技，而是让模型切换这件事有语义、有反馈、有边界。

### 3. 减少“黑箱感”

Yachiyo 后面会越来越有 runtime、memory、harness、plugin 这些东西。

但 UX 上不能让用户感受到一坨系统名词。

更好的方式是把中间过程翻译成用户能理解的轻量语言，比如：

- 正在准备回答
- 正在整理上下文
- 正在使用额外能力
- 已停止
- 生成失败

也就是说，用户感受到的是“过程”，不是“内部实现名词”。

### 4. 让结束状态更完整

一条回答的结束，不该只有“内容停了”。

至少应该区分出：

- 正常完成
- 失败
- 被停止

这三种收尾会直接影响用户对系统可靠性的判断。

尤其是 stop，如果用户主动点了停止，界面应该给出一种明确但不夸张的收束感，而不是像异常中断。

## 对 Claude 的实现导向

如果把这份文档 quote 给 Claude，核心意思应该是：

不要先从技术事件表出发，不要先从协议字段出发，也不要把精力放在过早定义很死的代码接口上。

先从 UX 出发，回答下面几个问题：

- 一次发送开始后，用户应该依次看到哪些状态
- 哪些状态需要进入消息区，哪些只适合放在全局位置
- model selector 怎样才能从“配置控件”变成“聊天上下文的一部分”
- 回答进行中、回答结束、回答失败、回答停止，这几种视觉和交互反馈要怎么区分
- 哪些中间过程值得给用户看，哪些不值得

重点是把已有能力翻译成体验，而不是新增一套面向工程师的可视化调试面板。

## 对当前界面的具体判断

### 1. `Thinking...` 太笼统

现在这种统一的 `Thinking...` 虽然有总比没有好，但它压掉了很多本来应该存在的交互层次。

它的问题不是文案本身，而是信息密度太低。

对用户来说，这种反馈无法回答：

- 系统只是刚收到消息，还是已经开始生成
- 系统是在单纯思考，还是正在处理额外能力
- 这次过程是否已经进入正文输出阶段

所以不一定要做很多状态，但至少要把单一的“Thinking”拆成更像聊天过程的几个轻阶段。

### 2. 底部状态条不该承担主要叙事

全局状态条可以保留，但它更适合承担：

- 服务不可用
- provider 未配置
- 明确错误

而不应该承担“这次回答正在发生什么”的主要表达。

真正与这次回答相关的过程反馈，更适合贴近消息本身。

不然用户的视线会在消息区和底部来回跳，体验会很割裂。

### 3. assistant message 现在太“只有正文”了

现在 assistant message 更像一个纯 markdown 容器。

但一个成熟一点的 chat 产品里，assistant message 往往还会带一点非常轻的上下文信息，比如：

- 当前模型
- 正在生成
- 已停止
- 失败

不是说要全部堆出来，而是 message 本身应该有能力承载这些状态。

### 4. 过程提示应该轻，不要喧宾夺主

如果要把中间阶段放进消息区，也不要做成大块的 debug card。

更合适的是一种轻量、低压迫感、不会抢正文注意力的表达。

它的作用只是告诉用户：

- 系统还在工作
- 当前阶段发生了变化
- 这次回答并不是凭空冒出来的

## 建议 Claude 优先做的三件事

### 第一件：重做 chat 的状态表达

先不要急着加新功能。

先把一条消息从发送到完成的体验节奏理顺。

重点是让用户能清楚感知：

- 开始了
- 正在处理中
- 正在输出
- 结束了

### 第二件：把 model selector 纳入消息体验

让用户能看见模型选择与当前回答之间的关系。

至少要让“选择了什么”和“实际用了什么”这两件事更清楚。

### 第三件：把中间过程做成轻量 UX，而不是技术名词展示

如果需要呈现过程，不要优先显示内部术语。

优先用更自然的、面向用户的语言去表达过程。

Yachiyo 的 UX 重点不是“暴露 runtime 细节”，而是“降低黑箱感，同时保持聊天的自然性”。

## 边界

这份 brief 刻意不把代码协议、事件映射、store 结构写得太死。

原因很简单：

- 当前阶段更需要的是交互方向正确
- 不是先把实现接口钉死
- Claude 应该根据现有代码结构和你要的 UX 方向，自行选择合适的实现方式

所以这份文档更像一个产品/体验导向的实现 brief，而不是后端协议规范。

## 最后的落点

Yachiyo 现在已经过了“证明它能跑”的阶段。

下一步最值钱的不是继续增加功能数量，而是让用户明显感受到：

- 这个 chat 是有节奏的
- 这个 chat 是有状态感的
- 这个 chat 对模型选择是有语义的
- 这个 chat 在处理中不是一团黑箱

如果这层 UX 被打磨出来，整个产品气质会立刻上一个台阶。
